# 📡 Qt Core: Signals & Slots Mastery (Official)

Type-safe, loosely-coupled communication. Signals and slots are **NOT** function pointers; they are a high-level abstraction built on top of the Meta-Object System.

---

## 1. 🛠️ The Syntax Evolution
Always use the **Pointer-to-Member-Function** syntax (Qt 5.0+).

```cpp
// Correct (Type-checked at compile time)
connect(sender, &QPushButton::clicked, receiver, &MyClass::onClicked);

// Lambda (Most powerful, no slot declaration needed)
connect(sender, &QPushButton::clicked, this, [=](bool checked) {
    qDebug() << "Checked status:" << checked;
});
```

## 2. ⛓️ Connection Types (The Threading Rules)
| Type | Behavior |
| :--- | :--- |
| `Qt::AutoConnection` | **Default.** Direct if same thread, Queued if different. |
| `Qt::DirectConnection` | Invoked immediately in the **sender's** thread. Dangerous if data isn't locked. |
| `Qt::QueuedConnection` | Invoked when control returns to the **receiver's** event loop. Thread-safe. |
| `Qt::BlockingQueuedConnection` | Sender blocks until the slot finishes. (Cross-thread only). |
| `Qt::SingleShotConnection` | Slot is called once, then automatically disconnected. |

## 3. ⚙️ Internal Management
- **`sender()`**: Inside a slot, returns `QObject*` of the sender. (Use sparingly; breaks decoupling).
- **`blockSignals(true)`**: Temporarily stop an object from emitting any signals.
- **`disconnect()`**: Break specific or all connections.

## 4. 🏁 The Logic Bridge
- **Signals** are function declarations generated by `moc`. You **never** implement them.
- **Slots** are standard C++ member functions.
- **Emitting:** The `emit` keyword is purely for the human reader; the compiler treats it as a normal function call to the `moc`-generated code.
