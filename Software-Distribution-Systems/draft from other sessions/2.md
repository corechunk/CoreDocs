### 1. FHS Binary Locations & Purposes

The Filesystem Hierarchy Standard (FHS) explicitly separates files based on who manages them (the OS package manager vs. the local administrator) and who can access them (all users vs. a specific user).

|**Binary Path**|**Scope**|**Managed By**|**Primary Purpose**|
|---|---|---|---|
|**`/bin`** (or `/usr/bin`)|System-Wide|Official Package Manager (`apt`, `pacman`)|Core, essential single-user or multi-user binaries required to boot or repair the system (e.g., `ls`, `bash`).|
|**`/sbin`** (or `/usr/sbin`)|System-Wide|Official Package Manager (`apt`, `pacman`)|Essential system administration binaries reserved for the root user (e.g., `iptables`, `fdisk`).|
|**`/usr/bin`**|System-Wide|Official Package Manager (`apt`, `pacman`)|The vast majority of user programs installed by the distribution (e.g., `gcc`, `ffmpeg`, your official `.deb`/`.pkg` app).|
|**`/usr/local/bin`**|System-Wide|Host Administrator (Manual or Installer Script)|Custom software compiled or dropped in locally by the admin. **Official package managers are forbidden from touching this path.**|
|**`/opt/`**|System-Wide|Third-Party Vendor (Monolithic Bundles)|Add-on application packages that contain their own isolated internal layout (e.g., `/opt/google/chrome/`).|
|**`~/.local/bin`**|User-Specific|Single User (No `sudo` required)|Local utilities, user-specific CLI tools, or scripts executed only within that specific user's login session.|

### 2. Which Distribution Method Targets Which Path?

If your project produces multiple distribution types, they must target completely separate paths to prevent breaking the system state.

- **Native Packages (`.deb`, `.pkg.tar.zst`, `.rpm`):** Must target **`/usr/bin`**. Because these files are managed by the official system database, they belong in the official system paths.
    
- **Root Installer Script (`sudo ./installer.sh`):** Must target **`/usr/local/bin`**. This signals to the system that a human administrator intentionally installed the file manually, leaving `/usr/bin` entirely clear for the package manager.
    
- **User Installer Script (`./installer.sh --user`):** Must target **`~/.local/bin`**. This ensures the application can install cleanly without requiring root permissions.
    
- **AppImage:** Does not target an installation directory path. It is a portable compressed filesystem. The user can download the file to `~/Applications/` or `/opt/`, mark it executable, and run it. At runtime, the AppImage dynamically mounts itself to a temporary folder under **`/tmp/`** and runs from there.
    

### 3. The Duplicate Execution & Shadowing Conflict

If a user ignores your advice and installs your tool using _both_ an installer script (targeting `/usr/local/bin`) and an `apt` package (targeting `/usr/bin`), it creates two identical executables in different locations.

When the user types your command into the terminal, the shell does not check both files or merge them. It reads the system's `$PATH` environment variable from left to right and executes the **first match** it finds.

A typical Linux `$PATH` looks like this:

Plaintext

```
~/.local/bin : /usr/local/bin : /usr/bin : /bin
```

#### The Result:

1. **Shadowing:** Because `/usr/local/bin` comes _before_ `/usr/bin` in the search order, the shell will execute the version installed by your script.
    
2. **The Broken State Trap:** If the user updates the application later via `apt update`, the new version is written to `/usr/bin`. However, because the old version from the script still lives in `/usr/local/bin`, the user will continue running the old version without realizing why their package updates are not working.
    

### 4. Handling Binary Name Collisions & Clean Aliasing

You cannot reliably edit a user’s `~/.bashrc` file or global `/etc/bash.bashrc` file during an automated installation pipeline. It is brittle, dirty, and non-standard.

You mentioned wanting a clean directory setup similar to `/etc/apt/sources.list.d/`. While `sources.list.d` is reserved strictly for network repository endpoints, the Linux shell architecture provides an exact programmable equivalent for custom configurations: **`/etc/profile.d/`**.

#### The Blueprint for Handling Name Collisions:

If your tool name conflicts with an existing binary, you should name your physical binary file something unique (e.g., `linutils-cli` or `linutils-tui`). You then manage shortcut access cleanly depending on your deployment path.

#### Method A: The System-Wide Drop-In Directory (`/etc/profile.d/`)

Every POSIX-compliant shell (Bash, Zsh, Sh) reads and executes every script inside `/etc/profile.d/` automatically upon user initialization.

When your installer runs with root permissions, it can drop a custom initialization file directly into this folder:

Bash

```
# Drop this file cleanly to /etc/profile.d/linutils.sh
alias lu='/usr/bin/linutils-cli'
```

The next time any user opens a terminal window, the `lu` alias is loaded automatically. To remove it during uninstallation, your script simply deletes that single file.

#### Method B: Symlink Aliasing (Cleanest Architectural Practice)

If you want users to have a shorter command name without using shell aliases, you can use filesystem symbolic links (`symlinks`).

If your core binary is officially named `/usr/bin/linutils-tool`, your installer script can place a localized symlink into the administrator's path:

Bash

```
ln -s /usr/bin/linutils-tool /usr/local/bin/lu
```

Because `/usr/local/bin` takes priority over `/usr/bin`, users can type `lu` to launch your tool instantly, while keeping your main binary isolated under its distinct unique name inside the package manager's files.
